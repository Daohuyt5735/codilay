## Overview
## Overview
The CodiLay project aims to enhance codebase management through dynamic, real-time documentation updates. Its architecture consists of several Python modules that work together to analyze, track, and update code documentation based on changes in the repository. The primary tech stack includes Python for backend operations, leveraging various libraries and services, and TypeScript for the VSCode extension.

### Key Components
- **Core Modules:** Located in `src/codilay/`, these include modules responsible for chunking files (`chunker.py`), managing configurations (`config.py`), tracking documents (`doc_differ.py`, `docstore.py`), and interfacing with version control systems (`git_tracker.py`).
- **Documentation Management:** The system uses a wire model to track unresolved references, allowing for dynamic documentation. `wire_manager.py` and `wire_bus.py` handle this aspect, ensuring that documentation remains updated as new code changes.
- **Integration with Language Models:** Modules like `llm_client.py` enable integration with LLMs to facilitate intelligent documentation suggestions.
- **Extension and Interface:** The VSCode extension (`vscode-extension/`) allows for interactive operation via a user-friendly interface.

### Entry Points
- **Command-Line Interface (CLI):** Executable via `src/codilay/__main__.py`, providing a command-line interface for key operations.
- **Web Server:** The FastAPI-based server (`server.py`) allows for API-based interactions and is also accessible via the command `codilay serve`.

### Developer Guidance
New developers should start by exploring `README.md` for an introduction to the system's capabilities and usage guidelines. Examine `contributing.md` for development and contribution directives. The configuration files (`codilay.config.json`, `pyproject.toml`) provide details on setup and dependencies.

## CodiLay File Chunker
> File: `src/codilay/chunker.py`

### Chunker Class (Part 1/2)
The `Chunker` class in the `chunker.py` module is responsible for dividing large files into processable chunks. It supports multiple programming languages by applying different strategies for each.

#### Initialization
- **`__init__(self, token_counter, config)`**: Initializes the `Chunker` instance with a token counter function and configuration settings. Key config parameters include `chunk_token_threshold`, `max_chunk_tokens`, and `chunk_overlap_ratio`. These dictate the chunking process, such as when splitting is needed and how chunks overlap.

#### Chunking Plan
- **`plan(self, file_path, content)`**: Analyzes a given file's content to determine a chunking strategy. Returns a `ChunkPlan` with unified or split chunks based on a token threshold. If content tokens exceed the threshold, the code distinguishes between a SKELETON chunk and multiple DETAIL chunks, each representing a segment of the file split defined by programming structures.

    - **Skeleton Extraction**: Calls `_extract_skeleton` to retrieve high-level elements like imports and function/class signatures, which are crucial for understanding the file's structure without delving into implementation details.

    - **Boundary Detection**: Determines where to split the file into chunks based on language-specific rules, such as class or function boundaries (implemented in methods like `_find_python_boundaries`).

#### Internal Workings
Methods like `_find_boundaries` and its language-specific counterparts (`_find_python_boundaries`, `_find_js_boundaries`, etc.) identify logical code segments or `FileChunk` instances. Each boundary or chunk is characterized by labels and symbols, detailing what each chunk represents or contains.

This class connects intricately with token counting mechanisms and configuration management (as seen in the reliance on a `CodiLayConfig` instance for settings), positioning it as a core component in managing codebase scalability within the system.

#### Detailed Chunk Processing
- **`_create_chunks(self, lines: List[str])`**: This method handles the actual breakdown of lines into `FileChunk` instances based on a token limit. When `current_tokens` exceed `max_chunk_tokens`, it attempts to split at a nearby blank line to maintain code readability.

    It manages a list of `chunks` starting from `current_start` and appends a new `FileChunk` object each time a chunk is complete. The chunk's label and line range are clearly defined, allowing for easy tracking during further processing or review. Remaining lines after a split continue building the next chunk.

    This method finalizes by adding any leftover lines as an additional chunk if necessary.

- **`_add_overlap(self, chunks: List[FileChunk], lines: List[str])`**: Implements context continuity by adding overlapping lines between chunks. Overlap is calculated as 10% of the previous chunk's lines (capped at 30 lines), providing context linkage to aid comprehensibility. This overlap is prefixed with clear markers indicating the borrowed context and new content, adjusting the start line accordingly.

- **`_sub_split_chunk(self, chunk: FileChunk)`**: Splits any oversized `FileChunk` into smaller sub-chunks, ensuring that each sub-chunk adheres to token constraints. Labels are updated to reflect their partial nature, maintaining continuity in reference throughout the process.

#### Helper Methods
- **File Syntax Identification**: Supports multiple languages by identifying key syntax types ŌĆö such as import lines, class/function signatures, comments, decorators, exports, and type/constants. Methods like `_is_import_line`, `_is_signature`, `_is_comment`, and others use pattern matching tailored to each language extension, facilitating chunk planning processes and ensuring language-specific nuances are respected.

    The `_count_body_lines` helper method counts lines of code in a function or class body using indentation for Python or curly braces for C-like languages, aiding in boundary detection.

## CodiLay File Scanner
> File: `src/codilay/scanner.py`

The `Scanner` class in `src/codilay/scanner.py` is responsible for scanning directories in a codebase while respecting ignore rules specified in `.gitignore` and additional configuration settings. It focuses on identifying text files that are relevant for further processing by the CodiLay system.

### Key Features

- **Ignore Rules:** Utilizes patterns from `.gitignore`, additional configuration settings, and other common ignore patterns to determine which files should be excluded from scanning. This ensures efficient processing by excluding unnecessary files and directories.

- **Text File Detection:** The scanner includes robust logic to identify 'text' files by their extensions or specific filenames. This helps in identifying files that are likely to contain code or documentation relevant to CodiLay's processing.

- **Git-aware Discovery:** The scanner uses `git ls-files` to efficiently and accurately discover files that are both tracked and untracked (but not ignored) by Git, potentially providing a comprehensive yet quick overview of the files in the repository.

- **File Tree Visualization:** The class can generate a structured text-based representation of the directory tree, which mirrors the file discovery logic, providing a visual representation of the codebase hierarchy after applying ignore rules.

- **Change Detection:** Integrates logic to determine changed files based on Git changes, allowing CodiLay to perform targeted documentation updates instead of processing entire file sets.

### Example Usage


## CodiLay Dependency Graph
> File: `src/codilay/dependency_graph.py`

The `DependencyGraph` class is a core component of CodiLay that enables the static analysis of file dependencies across various programming languages. This class builds a directed acyclic graph (DAG) from static import statements found in the codebase, allowing the identification of files that can run in parallel and those that require sequential execution.

#### Initialization
The `DependencyGraph` is initialized with a `project_root` and a list of `all_files` in the project. It maintains a file index for quick lookup and two core dictionaries: `depends_on` and `depended_by`. These track which files import from and are imported by others, respectively.

```
from codilay.dependency_graph import DependencyGraph

# Initialize the graph
graph = DependencyGraph(project_root="/path/to/project", all_files=["file1.py", "file2.py"])
```

#### Methods
- **build(file_contents):** Analyzes provided file contents to construct the dependency graph.
- **get_tiers():** Returns a list of processing tiers where each tier represents files that can be executed in parallel based on their dependencies (Kahn's algorithm is used for topological sorting).
- **get_dependency_clusters():** Identifies clusters of files that form connected components in the graph, indicating groups of files that have dependency relationships.
- **get_parallel_groups(tier):** Further subdivides a tier into parallel groups that share no wire targets, indicating true parallelism.
- **get_files_affected_by(file_path):** Determines all files that transitively rely on a specified file.
- **get_stats():** Provides statistics about the constructed graph such as the number of files, edges, tiers, and clusters.

#### Import Extraction
The class includes methods for extracting import statements from various languages, such as Python, JavaScript, Go, Rust, and more. These include:
- **_extract_imports(file_path, content):** General method that delegates to language-specific extractors.
- **_extract_python_imports(content):** Extracts import statements specifically from Python code.

#### Import Resolution
Resolves import strings to actual file paths in the project. It handles relative imports, direct paths, module-style imports, and basename matches via methods like:
- **_resolve_imports(source_file, raw_imports):** Attempts to resolve all imports for a source file.

This package component is essential for efficient and accurate parallel execution planning within the CodiLay system even in environments without LLM calls.

## CodiLay Graph Filter Functionality
> File: `src/codilay/graph_filter.py`

### Graph Filters

The `graph_filter.py` module provides a robust filtering mechanism to manage and visualize the dependency graph of a codebase. This system is particularly useful for handling large repositories where the quantity of dependencies can become overwhelming.

#### GraphFilterOptions

This data class defines the filters that can be applied to a dependency graph:
- **Wire Types:** Specifies the types of connections to consider, such as `import`, `call`, or `reference`.
- **File Layers:** Filters nodes and edges by file directory prefixes.
- **Modules:** Uses pattern matching to include or exclude modules.
- **Exclude Patterns:** Defines file path patterns to exclude from the graph.
- **Direction:** Determines whether to show `incoming`, `outgoing`, or `both` connections.
- **Depth Limit:** Limits graph traversal depth.
- **Minimum Connections:** Shows only nodes exceeding a certain connection count.

#### FilteredNode and FilteredEdge

`FilteredNode` and `FilteredEdge` are dataclasses representing nodes and edges in a filtered graph.
- **FilteredNode:** Contains path, label, inferred layer, and counts of incoming and outgoing connections.
- **FilteredEdge:** Contains source, target, wire type, summary, and wire ID.

#### FilteredGraph

The `FilteredGraph` class computes the result of applying filters and organizes nodes and edges accordingly. It has methods to convert the graph to a dictionary and explore available wire types and layers.

#### GraphFilter

The `GraphFilter` class processes closed and optionally open wires to provide filtering functionalities with precise methods for applying each filter, such as `_matches_layers`, `_matches_modules`, and `_matches_exclude`.

**Usage:** This filter system is employed in both CLI commands and web UI graph views, allowing users to customize their views to focus on segments of the graph pertinent to their development work.

**Examples:**
```python
filter_options = GraphFilterOptions(wire_types=['import'], layers=['src/models'])
filtered_graph = graph_filter.filter(filter_options)
print(filtered_graph.to_dict())
```

## Parallel Orchestrator File Processing
> File: `src/codilay/parallel_orchestrator.py`

The `ParallelOrchestrator` class in the `src/codilay/parallel_orchestrator.py` file coordinates parallel file processing based on dependency tiers using a thread-based approach.

#### Architecture & Components:
- **Central Wire Bus**: Acts as a shared and locked communication channel for worker threads to resolve or open wires.
- **Workers**: Independent threads responsible for processing files concurrently within a tier.
- **DocStore**: Manages documentation sections and provides locked access to ensure thread safety during updates.

#### Processing Flow:
1. **Dependency Graph Construction**: Utilizes static import analysis to build a graph of file dependencies.
2. **Tiers Calculation**: Topologically sorts files into execution tiers ensuring no intra-tier dependencies.
3. **Tier Execution**:
   - Files within a tier are processed in parallel using a thread pool.
   - Workers utilize frozen wire snapshots to prevent partial reads.
   - Results are written to the DocStore with lock-based synchronization.
   - Tier boundaries act as synchronization points for state reconciliation.
4. **Parked Files**: Files that can't proceed due to unresolved dependencies are parked and handled sequentially once dependencies are resolved.
5. **Finalization**: A sequential pass reviews sections generated in parallel and ensures consistency.

#### Key Classes & Data:
- `WorkerResult`: Captures the outcome of file processing, including success, errors, and wire operations.
- `TierResult`: Tracks the results and overall execution time for a tier.
- `ParkEntry`: Describes files that are temporarily parked due to unmet dependencies.

#### Synchronization & Safety:
- **Wire and DocStore Locks**: Ensure consistent updates across threads by securing concurrent operations.
- **Progress Tracking**: Incrementally reports status and updates the processing queue.

The `ParallelOrchestrator` ensures efficient documentation generation by leveraging parallel processing while maintaining data integrity through robust synchronization and locking mechanisms.

## CodiLay File Triage System
> File: `src/codilay/triage.py`

The `src/codilay/triage.py` module implements the Triage system for CodiLay, which classifies files into categories to optimize processing. It uses two modes of operation: fast and smart, tailored for differing levels of detail and computational overhead.

### Triage Options
- **Fast Mode**: Utilizes static patterns to quickly classify files without invoking a Language Learning Model (LLM). This mode instantly separates files into core, skim, and skip categories based on predefined patterns, hence being computationally efficient.
- **Smart Mode**: Employs an LLM to analyze the project structure, without sending file content, to make more informed decisions. This mode can adapt to complex projects by leveraging AI insights aggregated over broader patterns.

### Key Classes and Functions
- **`FileCategory`**: Specifies possible categorization outcomes for files.
- **`TriageResult`**: A data class representing the output of the triage process, containing lists of files for each category and summarizing the triage reasoning.
- **`Triage`**: The main class orchestrating the triage process through `fast_triage` and `smart_triage` methods.
  - **`fast_triage`**: Classifies files using static patterns, bypassing the need for LLM.
  - **`smart_triage`**: Engages the LLM to decide the classification, providing fallback to fast mode if needed.

### Pattern Configuration
This module defines static patterns for directories, file extensions, and filenames that guide the baseline triage decision in fast mode. These include:
- **`ALWAYS_SKIP_DIRS`**: Directories like `.git` and `node_modules` that are automatically omitted.
- **`ALWAYS_SKIP_EXTENSIONS`**: File extensions like `.pyc` that are not processed.
- **`ALWAYS_SKIM_FILES`**: Files like `package.json` and `Dockerfile` that usually require limited processing.

### Integration
The Triage module is a crucial initial step in the CodiLay documentation pipeline, determining which files require deep analysis and which are to be ignored or briefly reviewed, thus saving processing time and resources.

### Example Usage
```python
from codilay.triage import Triage

triage = Triage()
result = triage.fast_triage(['src/main.py', 'README.md', 'node_modules/library.js'])
print(result.core)  # ['src/main.py']
```
